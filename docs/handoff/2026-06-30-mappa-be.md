## 2. Task delegato: modulo `mappa` (modello + lettura)

**Decisione (secondo gli accordi scritti):** il prossimo slice è la **mappa**, non lo staff-mgmt (D-025). Motivi: è **Core MVP** (spec approvata 2026-06-27) mentre D-025 è *deferred* a impatto basso; è il **prerequisito architetturale** di prenotazioni e pricing; la **FE è già scritta** contro un **contratto già definito** (`MappaGiornoDTO`) ed è solo mockata → ROI massimo. Lo staff-mgmt/D-025 è l'increment **successivo**.

**Scope di QUESTO slice:** entità della mappa + RLS + seed di una struttura demo + endpoint di **sola lettura** `GET /api/mappa` + sgancio della `MappaView` dal mock. **Fuori scope** (increment successivi): il *setup-form* (CRUD di settori/file/ombrelloni, [ADR-0014](../architecture/decisions/0014-setup-mappa-strutturato.md)), le prenotazioni, il pricing.

### Modello dati (fonte di verità: [data-model.md](../design/data-model.md))

Aggiungi 5 modelli Prisma, **tutti tenant-scoped** (`stabilimentoId` + RLS come `Cliente`):

- **Settore**: `id`, `stabilimentoId`, `nome`, `ordine`.
- **Fila**: `id`, `stabilimentoId`, `settoreId`, `etichetta`, `ordine`.
- **Ombrellone**: `id`, `stabilimentoId`, `filaId`, `tipologiaId?` (nullable = Normale, ADR-0016), `etichetta` (**numero fisico reale, unico per Stabilimento** → unique composito `[stabilimentoId, etichetta]`), `ordineLogico`. (`posizionePresentazione` json nullable = layer visivo D-005: modellala nullable ma lasciala inutilizzata in questo slice.)
- **Tipologia**: `id`, `stabilimentoId`, `nome`, `ordine`, `icona?` (chiave registry icone ui-kit, ADR-0020).
- **Fascia**: `id`, `stabilimentoId`, `nome`, `oraInizio`, `oraFine`, `ordine`.

⚠️ Ogni nuova tabella deve **ENABLE RLS + policy `tenant_isolation`** come `Cliente` (vedi migrazione `20260628175658_rls`): `USING`/`WITH CHECK` su `nullif(current_setting('app.current_tenant', true), '')::uuid = "stabilimentoId"`. Scrivi la RLS come **SQL grezzo nella migrazione** (Prisma non la genera). Tutte le query passano da `PrismaService.forTenant`.

### Endpoint

`GET /api/mappa?data=YYYY-MM-DD` → `MappaGiornoDTO`:
- `data` opzionale (default: oggi), ISO `yyyy-mm-dd`; viene echeggiata nel DTO.
- Dentro `forTenant(tenantId)`: carica tipologie, fasce, settori→file→ombrelloni ordinati per `ordine`/`ordineLogico`; proietta nei DTO (`FasciaDTO` espone solo `id/nome/ordine`, **non** ora_inizio/fine).
- **`statoPerFascia`**: in questo slice (senza prenotazioni) ogni ombrellone è **`libero`** per ogni fascia. La derivazione reale (abbonato/giornaliero/prenotato) arriva con lo slice **prenotazioni** — è il confine d'incremento, dichiaralo nel codice/spec. **Non** inventare stati.
- `MappaModule` + `MappaController` + `MappaService`. Protetto da `JwtAuthGuard` (già globale).

### Seed (così la mappa non è vuota e la FE si sgancia davvero)

Estendi `apps/api/prisma/seed.ts` per creare, **idempotente**, per `DEV_STABILIMENTO_ID`: 2–3 tipologie (es. Mini-palma `leaf`, Palma `palmtree`), 2 fasce (Mattina/Pomeriggio), 2 settori (`Centro` + `Speciali`), qualche fila e ~20–40 ombrelloni con etichette reali. Riusa il pattern `upsert` con id stabili. La guardia `NODE_ENV=production` resta (l'entrypoint Docker seeda con `NODE_ENV=development`). Vedi `apps/web-staff/src/mocks/data/seed.ts` per la **forma attesa dalla FE**.

### La FE mockata è il contratto vincolante (non riprogettarla)

⚠️ **La `MappaView` esiste già, è completa e renderizza `MappaGiornoDTO` dinamicamente.** È **il backend che deve adattarsi alla FE**, non viceversa. Il file `apps/web-staff/src/mocks/data/seed.ts` (`mappaSeed`) è la **forma di riferimento** e va **conservato come fixture di test** (non eliminato). Vincoli di correttezza che ne derivano:

- L'output di `GET /api/mappa` deve **combaciare 1:1** con `MappaGiornoDTO` (`data`, `tipologie[]`, `fasce[]`, `settori[]→file[]→ombrelloni[]`). La FE non hardcoda id → lato server usa **UUID** (nessun accoppiamento agli id del mock come `o-1`/`s-centro`); conta solo la **forma**.
- **Le chiavi di `OmbrelloneDTO.statoPerFascia` devono essere esattamente gli `id` delle `FasciaDTO` ritornate** nello stesso payload (la FE indicizza lo stato per `fascia.id`). Ogni ombrellone deve avere una entry per **ogni** fascia.
- `TipologiaDTO.icona` deve essere una **chiave valida del registry icone** del `ui-kit` (ADR-0020) — verifica in `packages/ui-kit/src/icons/registry.ts` (il mock usa `leaf`/`palmtree`); un'icona inesistente rompe il marker.
- `Ombrellone.tipologiaId` nullable = Normale (ADR-0016); il settore "Speciali" (palme) ha `ordine` alto (il mock usa 99). Replica una struttura demo coerente col mock così la vista renderizza come oggi.

**Sgancio operativo (dev usa il backend reale, test restano deterministici):** oggi `apps/web-staff/src/mocks/handlers.ts` mocka `GET /api/mappa` **sia in dev** (worker MSW in `main.ts`) **sia nei test** (`server.ts` fa `setupServer(...handlers, ...)`). **Sposta** il mock `/api/mappa` da `handlers.ts` (dev) dentro `server.ts` (solo test), riusando `mappaSeed` come fixture. Così il dev worker cade sul backend reale (`onUnhandledRequest: 'bypass'`) e `MappaView.spec.ts` resta verde sulla fixture.

- In `useMappaGiorno.ts` passa la data attiva (TODO già annotato): `apiFetch<MappaGiornoDTO>(`/mappa?data=${data}`)` con `session.dataAttiva`.
- I contratti esistono già: **non** rinominare/rimuovere export di `@coralyn/contracts` (solo additivo se strettamente necessario, additivo e condiviso FE/BE).

### Workflow atteso (come gli increment precedenti)

Segui la prassi del progetto ([ADR-0009](../architecture/decisions/0009-documentazione-di-design.md)): prima una **breve design spec** del backend mappa (endpoint, proiezione DTO, RLS, seed) in `docs/specs/`, poi un **piano TDD** in `docs/plans/`, poi implementazione **test-first**. ## 4. Insidie note (dalle sessioni precedenti)

- **`prisma generate` PRIMA di `nest build`** (il Dockerfile lo fa già; in locale fallo dopo ogni cambio schema, altrimenti `TS7006`/tipi mancanti).
- **RLS sulle nuove tabelle**: senza `ENABLE RLS` + policy nella migrazione, o senza `forTenant`, hai leak o blocchi silenziosi. Ricorda la GUC `app.current_tenant`.
- **`prisma migrate dev`** usa lo shadow DB → serve `CREATEDB` su `coralyn_app` (ce l'ha, [D-023](../architecture/deferred.md)). In locale il **DB è su host `5433`** (override gitignored; la 5432 è occupata). Dev: `pnpm dlx dotenv-cli -e .env -- pnpm --filter @coralyn/api exec prisma migrate dev --name mappa`.
- **e2e**: girano contro **`coralyn_test`** via `.env.test` **locale gitignored** (ricrealo: `DATABASE_URL=postgresql://coralyn_app:coralyn_app@localhost:5433/coralyn_test?schema=public`, più `JWT_SECRET`/`JWT_EXPIRES_IN`). Applica le migrazioni a `coralyn_test` con `migrate deploy` prima degli e2e.
- **Seed** idempotente; la guardia `NODE_ENV=production` blocca il seed → l'entrypoint forza `NODE_ENV=development`.
- **pnpm via corepack** pinnato `pnpm@11.9.0`; per install non-interattivo usa `CI=true corepack pnpm install` (la rinomina di package può richiedere il purge di `node_modules`).
- **Dev server stantio**: dopo un reinstall, **killa il vecchio `pnpm dev`** (processo `node`, non "vite") prima di riavviare, altrimenti la risoluzione moduli stantia dà errori fantasma (es. `ENOENT … @coralyn/ui-kit/styles/theme.css`). Pulisci `node_modules/.vite` se serve.
- **Sweep/verifica nomi**: usa `rg -uu` (ripgrep salta le dot-dir come `.claude/` di default).
- `.gitattributes` forza **LF** su `*.sh`/`Dockerfile`/`nginx.conf`. **Git Bash storpia i path di `docker exec -w`** → usa PowerShell o `MSYS_NO_PATHCONV=1`.
- Auth mockata **solo** nei test; in dev il login va al backend reale via proxy Vite (`/api`→`:3000`).
